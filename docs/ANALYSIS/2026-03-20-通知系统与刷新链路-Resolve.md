# Resolve 文档

## 文档目的

本文档用于给出当前通知系统与 Token 刷新问题的统一实现方案，并与以下 BUG 文档对应：

- `docs/BUG/2026-03-20-IMAP-Refresh-Token-Bug.md`

目标不是继续讨论方向，而是把已经确认的产品口径收敛成可实施的落地方案。

---

## 已确认的产品口径

### 1. 邮箱页只负责“开不开通知”

每个邮箱只保留一个统一通知开关：

- 开启通知：该邮箱允许参与自动通知
- 关闭通知：该邮箱无论什么渠道都不发送

邮箱页不再分别配置：

- Telegram 是否发送
- Email 是否发送

### 2. 用户设置页负责“渠道配置”

通知渠道是否真正生效，统一由设置页决定：

- Telegram 通道是否启用
- Email 通道是否启用
- Telegram 的配置是否完整可用
- Email 的配置是否完整可用

### 3. 最终发送规则

自动通知是否发送，按以下规则计算：

- 邮箱已开启通知
- 且某渠道在设置页中已启用
- 且该渠道配置完整可用

因此最终效果为：

- 邮箱开启通知，设置页只启用 Telegram，则只发 Telegram
- 邮箱开启通知，设置页只启用 Email，则只发 Email
- 邮箱开启通知，设置页同时启用 Telegram 和 Email，则双发
- 邮箱未开启通知，则两边都不发

### 4. Token 刷新只适用于 Outlook

Refresh Token 刷新只适用于 Outlook / OAuth 账号：

- `account_type = 'outlook'` 才允许刷新
- `account_type = 'imap'` 不参与单个刷新、全量刷新、手动触发定时刷新

## 当前系统存在的问题

### 问题一：通知入口设计不一致

当前系统中：

- 设置页存在全局 `email_notification_enabled`
- 账号页存在 `telegram_push_enabled`
- 两者语义不统一

造成的结果是：

- 用户无法直观理解“邮箱开关”和“渠道开关”的边界
- 逻辑实现也容易跑偏

### 问题二：Email 自动转发当前不会执行

当前调度器被切成只运行 Telegram Job，导致 Email 自动通知链路虽然代码存在，但后台没有真正执行。

### 问题三：Email 渠道仍然是全局逻辑

`notification_dispatch` 当前对 Email 的判断仍然是：

- 只要全局 `email_notification_enabled = true`
- 就对所有 source 都挂上 Email channel

这不符合已确认的产品口径。

### 问题四：Token 刷新链路实现不一致

后台定时刷新已经只处理 Outlook，但手动刷新相关接口仍会把 IMAP 带进去，导致行为不一致。

---

## 统一实现方案

## 一、数据模型调整

### 账号表

保留一个统一账号级通知开关：

- 推荐新增字段：`notification_enabled`

建议处理方式：

- 短期兼容：用现有 `telegram_push_enabled` 迁移为统一通知开关语义
- 中期收口：代码中逐步改名为 `notification_enabled`

理由：

- 现有账号页的推送按钮本质上就是“这个邮箱是否参与通知”
- 只是现在字段名带着 `telegram`，容易误导

### 设置表

用户设置页保留并明确两个渠道级配置：

- Telegram 通道
  - `telegram_enabled`
  - `telegram_bot_token`
  - `telegram_chat_id`
  - `telegram_poll_interval`
- Email 通道
  - `email_notification_enabled`
  - `email_notification_recipient`

说明：

- `email_notification_enabled` 在新语义下不再表示“所有邮箱都自动发”
- 它表示“Email 通道已启用，可参与通知分发”

建议新增：

- `telegram_enabled`

理由：

- 当前 Telegram 是通过“Token / Chat ID 是否填写”隐式判断
- 最好改成明确开关，便于与 Email 保持一致

---

## 二、调度器调整

### 目标

恢复统一通知分发 Job，不再让 Telegram 和 Email 走两套互斥调度。

### 调整原则

- 调度器只保留一个“统一通知分发 Job”
- 该 Job 内部决定：
  - 哪些邮箱参与通知
  - 哪些渠道要发送

### 建议改法

在 `outlook_web/services/scheduler.py` 中：

1. `configure_scheduler_jobs(...)`
   - 改回调用 `_configure_email_notification_job(...)`
   - 但建议将其重命名为更准确的名字，例如：
     - `_configure_notification_dispatch_job(...)`

2. Job 名称与文案也一起收口：
   - 中文：`统一通知分发`
   - 英文内部语义：`Notification Dispatch`

### 不建议继续保留的结构

- “启用 Telegram 就跑 telegram_push_job”
- “启用 Email 就跑 email_notification_job”

这会继续制造双轨逻辑，后面还会再出问题。

---

## 三、通知分发逻辑调整

### 当前应保留的核心能力

`outlook_web/services/notification_dispatch.py` 已经具备这些重要能力：

- 拉取消息
- 游标管理
- 去重投递日志
- 多渠道发送抽象

这些能力应继续复用，不需要推翻。

### 需要修改的核心判断

当前 `_build_active_channels_for_source(...)` 的 Email 逻辑是全局挂载，需要改成：

#### 先判断邮箱是否参与通知

对普通邮箱：

- 读取账号级 `notification_enabled`

对临时邮箱：

- 可以先沿用“默认参与”或后续单独设计
- 当前建议先不在本轮复杂化，普通邮箱先统一收口

#### 再判断各渠道是否启用

如果邮箱参与通知，则：

- Telegram 通道启用且配置可用时，加入 Telegram channel
- Email 通道启用且配置可用时，加入 Email channel

### 推荐的伪代码

```python
if not mailbox_notification_enabled:
    return []

channels = []

if telegram_channel_enabled and telegram_runtime_ready:
    channels.append(TELEGRAM)

if email_channel_enabled and email_runtime_ready:
    channels.append(EMAIL)

return channels
```

### 结果

这样统一通知分发逻辑就和产品语义完全一致。

## 四、账号接口调整

### 当前状态

当前账号页使用：

- `telegram_push_enabled`
- `/api/accounts/<id>/telegram-toggle`

### 建议调整

将其产品语义改成统一通知开关：

- 字段对外返回改成：`notification_enabled`
- 接口改成：
  - `/api/accounts/<id>/notification-toggle`

### 兼容策略

为降低一次性改动风险，建议分两步：

#### 第一步

- 后端先保留 DB 字段 `telegram_push_enabled`
- 但 API 层改名映射为 `notification_enabled`
- `telegram-toggle` 路由先兼容存在，但内部语义改为统一通知开关

#### 第二步

- 前端全部替换后，再考虑数据库字段重命名

## 五、设置页调整

### 当前问题

当前设置页文案会误导用户：

- “启用邮件通知”看起来像全局对所有邮箱生效
- “Telegram 推送”看起来像邮箱页独立能力

### 建议文案

#### Telegram 区块

- 中文：`Telegram 通知`
- 英文：`Telegram Notifications`

辅助提示：

- 中文：`配置 Telegram 通知通道。只有开启通知的邮箱才会通过该通道发送。`
- 英文：`Configure the Telegram notification channel. Only mailboxes with notifications enabled will send through this channel.`

#### Email 区块

- 中文：`Email 通知`
- 英文：`Email Notifications`

辅助提示：

- 中文：`配置 Email 通知通道。只有开启通知的邮箱才会通过该通道发送。`
- 英文：`Configure the email notification channel. Only mailboxes with notifications enabled will send through this channel.`

#### 邮箱页

- 中文：`开启通知`
- 英文：`Enable Notifications`

辅助提示：

- 中文：`该邮箱是否参与自动通知；具体发送渠道由当前设置决定。`
- 英文：`Whether this mailbox participates in automatic notifications. Delivery channels are determined by the current settings.`

## 六、Token 刷新修复方案

### 目标

统一所有刷新入口，只允许 Outlook 账号进入 Token 刷新逻辑。

### 需要修复的路径

- `outlook_web/controllers/accounts.py`
  - `api_refresh_account(...)`
- `outlook_web/services/refresh.py`
  - `stream_refresh_all_accounts(...)`
- `outlook_web/services/refresh.py`
  - `stream_trigger_scheduled_refresh(...)`

### 修复原则

#### 单个账号刷新

- 读取 `account_type`
- 如果是 `imap`：
  - 直接返回失败
  - 提示：`IMAP 账号不支持 Token 刷新`

#### 全量刷新 / 手动触发定时刷新

- 查询账号时只筛 `account_type = 'outlook' OR account_type IS NULL`

这样与 `scheduled_refresh_task(...)` 逻辑保持一致。

## 七、实施顺序建议

### Phase 1：先修逻辑，不动太多 UI

1. 修复 Token 刷新过滤
2. 恢复统一通知分发 Job
3. 让 Email 渠道按“邮箱是否开启通知 + 设置页渠道开关”工作
4. 保持旧字段兼容

目标：

- 先把行为修对

### Phase 2：再收口前端与接口命名

1. 账号页“Telegram 推送”改成“开启通知”
2. 设置页文案改成渠道配置语义
3. API 返回字段从 `telegram_push_enabled` 逐步迁移为 `notification_enabled`
4. i18n 中英文词条统一更新

目标：

- 再把表达修对

## 八、测试建议

### 后端测试

新增或调整测试覆盖以下场景：

1. 邮箱关闭通知，Telegram / Email 都不发送
2. 邮箱开启通知，设置页只启用 Telegram，只发 Telegram
3. 邮箱开启通知，设置页只启用 Email，只发 Email
4. 邮箱开启通知，设置页双通道启用，双发
5. Email 通道未配置完整时，不发送 Email 但不影响 Telegram
6. Telegram 通道未配置完整时，不发送 Telegram 但不影响 Email
7. IMAP 账号单个刷新被拒绝
8. 全量刷新跳过 IMAP
9. 手动触发定时刷新跳过 IMAP

### 前端验证

1. 账号页开关文案与行为一致
2. 设置页 Telegram / Email 文案与行为一致
3. 中文英文切换后文案保持一致语义
4. 保存后重新加载设置，状态回显正确

## 九、结论

本轮不建议继续沿用“Telegram 开关”和“Email 全局开关”拼接出来的旧结构。

应统一收口为：

- 邮箱页只决定“这个邮箱是否参与通知”
- 设置页只决定“系统当前启用了哪些通知渠道”
- 统一通知分发负责根据这两个条件组合决定最终投递
- Token 刷新统一只服务于 Outlook / OAuth 账号

这是当前最简单、最符合用户理解、也最容易稳定维护的实现方案。
