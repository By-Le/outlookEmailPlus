# === 系统提示词中关于 ask_user 工具的完整说明 ===

## 1. 工具定义部分（function schema）

工具名称：ask_user
描述：Ask the user a question and wait for their response.
用途：Use this tool when you need to ask the user questions during execution.

功能：
1. Gather user preferences or requirements
2. Clarify ambiguous instructions
3. Get decisions on implementation choices as you work
4. Offer choices to the user about what direction to take

参数：
- question: 要询问用户的问题
- choices: 可选的多选项数组
- allow_freeform: 是否允许自由文本输入（默认 true）

---

## 2. 使用指南部分（<ask_user> 标签内）

**IMPORTANT: Never ask questions via plain text output.**
当你需要用户输入时，使用此工具而不是在响应文本中提问。

Guidelines:
- Prefer multiple choice (provide choices array) over freeform for faster UX
- Do NOT include "Other", "Something else", or similar catch-all choices
  （UI 会自动添加自由输入选项）
- Only use pure freeform (no choices) when the answer truly cannot be predicted
- Ask one question at a time - do not bundle multiple questions together
- Don't ask the questions in bullet points or numbered lists. 
  Ask each question in a clear sentence or paragraph form.
- If you recommend a specific option, make that the first choice and add "(Recommended)" 
  to the label

Examples of BAD usage:
1. 将多个问题打包成一个，让用户选择是否逐个讨论
2. 在问题文本中嵌入选项而不使用 choices 字段

Examples of GOOD usage:
1. 每次只问一个聚焦的问题
2. 将选项放在 choices 数组中

**When to STOP and ask (do not assume):**
- Design decisions that significantly affect implementation approach
- Behavioral questions (e.g., "should this be unlimited or capped?")
- Scope ambiguity (e.g., which features to include/exclude)
- Edge cases where multiple reasonable approaches exist

---

## 3. 关键要点

❌ 系统提示词中 **没有** 说：
- "任务完成后就不用提问"
- "只在需要澄清时才使用 ask_user"
- "任务已经回答清楚就直接结束"

✅ 系统提示词中 **实际说的是**：
- 当需要提问时，必须用 ask_user 工具（不能用纯文本）
- 列举了适合提问的典型场景（设计决策、行为选择、范围歧义、边界情况）
- 这些是"何时应该提问"的建议，而非"何时禁止提问"的限制

---

## 4. 与您的 Hook 规则的关系

您的 Hook 规则：
"无论当前任务是什么，无论任务是否完成，无论回复长短，
 只要准备结束回复就必须先使用 askQuestions 工具向用户提问"

这是对系统默认行为的 **强化扩展**，而非"与系统提示词冲突"。

系统提示词只是说明了 ask_user 的典型用途，
从未禁止在任务完成后使用它。

您的规则将"可选的最佳实践"升级为"强制的结束流程"。
